# Observability Stack on Azure (AKS)

Full observability stack deployed on **Azure Kubernetes Service**.

**Infra (Azure):** Plain Terraform
**Apps (K8s):** Terragrunt (Helm charts)

> **Note:** PostgreSQL disabled -- the eastus2 region does not support PostgreSQL Flexible Server in this subscription. The focus is only on AKS + monitoring stack.

Pattern: [aws/eks_rds](../aws/eks_rds/) -- infra with plain Terraform, K8s apps with Terragrunt.

## Deploy

### 1. Bootstrap (one time only)

Creates remote state backend: Resource Group + Storage Account + Container.

```bash
cd bootstrap
terraform init
terraform apply -auto-approve
```

### 2. Infra Azure (AKS via Terraform)

```bash
cd deploy/environments/dev/tf
terraform init
terraform apply
```

> AKS takes ~15 min to create.

> After apply, update kubeconfig with admin credentials (required for Terragrunt to connect):
> ```bash
> bash ../../../modules/aks/update-kubeconfig.sh
> ```

### 3. Monitoring (Helm charts via Terragrunt)

Deploy all components:

```bash
cd deploy/environments/dev/monitoring
find . -name ".terragrunt-cache" -type d -exec rm -rf {} + 2>/dev/null
terragrunt run --all apply
```

### 4. Access the cluster

```bash
kubectl get nodes
kubectl get pods -n monitoring
```


## Technical Decisions

### root.hcl instead of terragrunt.hcl

Newer Terragrunt versions (v0.67+) deprecate `terragrunt.hcl` as the root config file. Using `terragrunt.hcl` at the root of a Terragrunt project is now considered an anti-pattern and will error in future releases. The file was renamed to `root.hcl`, which is the recommended convention. All child modules use `include "root" { path = find_in_parent_folders() }` to locate it automatically.

### Provider generation with `generate` block

The `root.hcl` uses a `generate "k8s_provider"` block to create a `k8s-provider.tf` file dynamically. This avoids hardcoding provider configuration in every child module. The generated file configures both `kubernetes` and `helm` providers with `config_path` pointing to the kubeconfig file generated by the AKS module.

The `helm` provider requires `kubernetes = { ... }` syntax (assignment with `=`) rather than `kubernetes { ... }` (block syntax) in the current provider version. This is a known breaking change in the Helm provider v3.2.0.

### Kubeconfig with admin credentials

The AKS `kube_config_raw` output uses Azure AD authentication, which requires `kubelogin` to be installed. However, the Terragrunt/Helm provider still fails because the kubeconfig contains device-code auth that cannot be resolved non-interactively. The solution is to overwrite the generated kubeconfig with `--admin` credentials (which bypasses Azure RBAC). An `update-kubeconfig.sh` script is auto-generated by the AKS module for this purpose.

### Dependency resolution with mock_outputs

When running `terragrunt run --all apply`, all modules execute in parallel. Some components depend on outputs from upstream modules (e.g., Grafana depends on Tempo). If the upstream module has no Terraform outputs defined (like `tempo` and `cert-manager`), Terragrunt cannot resolve the dependency and fails with:

```
resolving dependency "tempo" outputs: detected no outputs
```

The fix is to add `mock_outputs` blocks in the dependent module's `terragrunt.hcl`:

```hcl
dependency "tempo" {
  config_path = "../tempo"
  mock_outputs = {
    tempo_service_name = "tempo"
    tempo_namespace    = "monitoring"
  }
}
```

Modules affected: `grafana`, `opentelemetry-collector`, `opentelemetry-operator`, `alertmanager`, `opencost`, `promtail`.

### Source path convention

All Terragrunt modules use the same relative path pattern:

```hcl
source = "../../../../../../../monitoring/modules/monitoring/<component>"
```

The depth (6 levels) matches the directory structure: `deploy/environments/dev/monitoring/<component>` to the root, then into the external `monitoring` repo.

### PostgreSQL disabled

The `eastus2` region does not support PostgreSQL Flexible Server in this Azure subscription. The PostgreSQL module, its subnets, and private endpoints were removed. The project focuses solely on AKS + monitoring.

## Alerts

| Alert | Severity | Description |
|-------|----------|-------------|
| PodCrashLoopBackOff | critical | Pod in crash loop |
| PodRestartHigh | warning | >3 restarts in 30min |
| PodNotReady | critical | Pod not ready for 5min |
| NodeDiskSpace | critical | Node disk <20% |
| NodeNotReady | critical | Node not ready |
| HighRequestLatency | warning | P95 latency >500ms |
| HighErrorRate | critical | HTTP 5xx >5% |
| PrometheusTargetDown | critical | Scrape target unreachable |

## Notifications

Alerts routed via **Alertmanager**:
- **Critical**: Email + Telegram
- **Warning**: Email only
- **Resolved**: Notified with status
